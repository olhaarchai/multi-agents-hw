# RAG для великих документів: який підхід найкраще підходить?

## Проблема великих документів

Великі документи створюють унікальні виклики для RAG:

- **Фрагментація контексту**: звичайне розбиття втрачає зв'язки між ідеями, розділеними великою відстанню
- **Контекстна дилема**: чанки повинні бути малими для точного пошуку, але достатньо великими для розуміння LLM
- **Напруга між витратами та якістю**: більші контексти коштують дорожче й гірше обробляються

---

## Порівняння підходів для великих документів

### 1️⃣ Naive RAG — НЕ рекомендується
**Чому не підходить:**
- Фіксовані розміри чанків не адаптуються до структури документа
- Втрачаються відносини між темами, розділеними далеко в документі
- Погано справляється з багатотемними великими корпусами

---

### 2️⃣ Sentence-Window — добре для середніх документів
**Як працює:**
- Документ розбивається на окремі речення
- При пошуку повертаються сусідні речення (вікно 2-5 речень)

**Переваги для великих документів:**
- ✅ Точна семантична відповідність (поліпшення на 22-38%)
- ✅ Гнучке налаштування розміру вікна
- ✅ Краще в локальному контексті

**Обмеження:**
- ❌ **Помиття контексту на дальні відстані**: якщо важлива інформація далеко, вікно її не охопить
- ❌ **Ростучі витрати**: великі вікна → більше токенів → вища вартість
- ❌ **Втрата глобальної структури**: не розуміє загальну архітектуру документа

---

### 3️⃣ Parent-Child (Hierarchical) — ОПТИМАЛЬНО для великих документів ⭐
**Як працює:**
- **Дочірні чанки** (200-400 токенів): конкретні ідеї, вкладаються й шукаються
- **Батьківські сегменти** (800-1200 токенів): більші контекстні блоки, передаються в LLM

**Переваги для великих документів:**
- ✅ **Збереження глобального контексту**: батьківські вузли утримують повну структуру теми
- ✅ **Ефективний пошук на дальні відстані**: може знайти пов'язану інформацію навіть далеко в документі
- ✅ **Оптимізована вартість**: вкладення лише для малих чанків
- ✅ **Точні цитування**: відслідкування до вихідного сегменту
- ✅ **Багатотемні запити**: добре впоратися з запитами, які потребують інформації з кількох місць в документі

**Сценарії, де Parent-Child найкраще:**
- Довгі вебінари (2+ години)
- Наукові монографії (200+ сторінок)
- Юридичні контракти та нормативні документи
- Технічна документація з множиною розділів
- Історичні дослідження з хронологічною еволюцією

---

## Порівняння на прикладі

**Запит**: "Як підхід до X еволюціонував протягом документа?"

| Підхід | Результат |
|--------|-----------|
| **Naive RAG** | Повертає розрізнені чанки; важко відстежити еволюцію |
| **Sentence-Window** | Добре знаходить локальні деталі; пропускає, коли ідеї далеко |
| **Parent-Child** | 🎯 Знаходить контекстні зв'язки на дальні відстані |

---

## Рекомендації для великих документів

### Розмір документа | Рекомендований підхід
- **< 5 MB** → Sentence-Window (простіше)
- **5-50 MB** → Parent-Child з ієрархією 2-3 рівнів
- **> 50 MB** → Parent-Child з ієрархією 3+ рівнів + recursive chunking

### Додаткові beste practices:
1. **Семантичне розбиття**: розділяйте документи за границями ідей, не за довільною кількістю токенів
2. **Рекурсивний підхід**: спочатку розділіть на розділи → параграфи → речення
3. **Metadata**: зберігайте інформацію про розділи, сторінки, дати для фільтрування
4. **Динамичне налаштування**: розмір батьківського чанка залежить від складності вмісту

---

## Висновок

Для **великих документів (> 10 MB або 5000+ сторінок)** **Parent-Child Retrieval** — оптимальний вибір, оскільки:
- Збереження історичної й контекстної целісності
- Економічність (вкладення лише для малих чанків)
- Точність пошуку на дальні відстани
- Можливість атрибуції й цитування джерел

**Sentence-Window** краще для документів середнього розміру (10-50 сторінок) де локальний контекст достатній.
