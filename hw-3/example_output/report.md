# Порівняння трьох підходів до побудови RAG: Naive, Sentence-Window та Parent-Child Retrieval

## 1. Naive RAG — Базовий підхід

**Характеристики:**
- Найпростіша й найрозповсюдженіша реалізація
- Процес: індексування → пошук → доповнення → генерація
- Документи розбиваються на фіксовані чанки, які перетворюються у вектори
- При запиті вибираються топ-K найбільш подібних чанків і передаються в LLM

**Переваги:**
- Мінімальна складність реалізації
- Низькі обчислювальні витрати
- Належний стартовий варіант для більшості проектів

**Недоліки:**
- Втрата контексту на границях чанків
- Фрагментація наративної структури документів
- Недостатня точність при складних запитах

---

## 2. Sentence-Window Retrieval — Оптимізація контексту

**Характеристики:**
- Документи розбиваються на **окремі речення**
- Кожне речення є окремим вузлом з ID та вкладенням
- Сусідні речення зберігаються як метаінформація (контекстне "вікно")
- При пошуку повертаються не лише знайдене речення, але й заздалегідь визначена кількість сусідніх речень

**Переваги:**
- **22.7% поліпшення Answer Relevance** порівняно з Naive RAG
- **38.2% поліпшення Groundedness** (менше галюцинацій)
- Більш точна семантична відповідність запиту
- Гнучке налаштування розміру вікна для різних типів вмісту

**Недоліки:**
- Більше вкладень в БД (більша кількість вузлів)
- Складніше налаштування гіперпараметру розміру вікна
- Вищі витрати на зберігання та обробку

---

## 3. Parent-Child Retrieval — Ієрархічний підхід

**Характеристики:**
- **Дво-рівнева структура:**
  - **Дочірні чанки** (200-400 токенів): малі, точні, для пошуку
  - **Батьківські сегменти** (800-1200 токенів): більші, з контекстом
- Вкладення генеруються лише для дочірніх чанків
- При пошуку повертається батьківський сегмент для генерації

**Переваги:**
- Збереження наративної структури документа
- **Можливість відповідати на складні запити** про еволюцію тем
- Точні цитування з залученням джерел
- **60% скорочення часу підготовки** (за даними кейсу)
- Краще збереження зв'язків між ідеями

**Недоліки:**
- Найбільша складність реалізації
- Потребує ретельного проектування ієрархії чанків
- Більші питання про витрати при дуже великих корпусах

## Рекомендації для вибору

- **Naive RAG**: для простих корпусів, прототипування, ресурсо-обмежених сценаріїв
- **Sentence-Window**: для збалансованого піджіння — більша точність без надмірної складності
- **Parent-Child**: для корпоративних систем, багатотемних документів, випадків де потрібна точна атрибуція
