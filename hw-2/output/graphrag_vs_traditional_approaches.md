# GraphRAG vs Традиційні Підходи до RAG: Детальне Порівняння

## Огляд: Четири Архітектури

Ми розглядаємо **чотири різні стратегії** побудови RAG систем, кожна з яких займає свою нішу в спектрі складності/якості:

```
Простота →  Naive RAG → Sentence-Window → Parent-Child → GraphRAG ← Складність
Якість  →   Низька      Середня           Висока        Найвисша
```

---

## 1. NAIVE RAG vs Інші Підходи

### Механізм Роботи
- Розбиває документи на **фіксовані за розміром чанки** (наприклад, 512 символів)
- Кожний чанк переводиться у вектор через embedding модель
- Пошук здійснюється через **семантичну подібність векторів**
- Знайдені чанки передаються прямо до LLM

### Характеристики
- **Структура даних**: Плоска векторна база даних
- **Тип пошуку**: Single-hop (один стрибок)
- **Розуміння зв'язків**: Відсутнє
- **Контекст**: Часто фрагментований

### Приклад
```
Запит: "Як зміна курсу валют впливає на ціни?"

Naive RAG знайде:
- Чанк про курси валют
- Чанк про ціноутворення
- Але не зрозуміє їх взаємозв'язок
```

---

## 2. SENTENCE-WINDOW RETRIEVAL vs Інші

### Механізм Роботи
- Розбиває документ на **речення** (семантичні одиниці)
- Обчислює вектори для **окремих речень**
- Зберігає розширене **контекстне вікно** (1-2 речення до/після основного)
- При пошуку повертає обширний контекст навколо релевантного речення

### Характеристики
- **Структура даних**: Векторна база + контекстні вікна
- **Тип пошуку**: Single-hop з розширеним контекстом
- **Розуміння зв'язків**: Локальне (тільки близькі речення)
- **Контекст**: Збережено в межах вікна

### Приклад
```
Запит: "Як зміна курсу валют впливає на ціни?"

Sentence-Window знайде:
- Речення про курси валют ✓
- Контекстне вікно навколо нього (попереднє та наступні речення) ✓
- Краще розуміння локального контексту
- Але все ще не побачить зв'язок з ціноутворенням у сусідньому параграфі
```

---

## 3. PARENT-CHILD RETRIEVAL vs Інші

### Механізм Роботи
- Створює **ієрархічну двошарову структуру**:
  - **Батьківські чанки**: Великі контекстні блоки (параграфи, розділи)
  - **Дочірні чанки**: Дрібні, точні фрагменти (речення)
- Вкладення обчислюється лише для **дочірніх чанків**
- Пошук знаходить дочірні чанки, повертає батьківські

### Характеристики
- **Структура даних**: Векторна база з ієрархією батьків-дітей
- **Тип пошуку**: Single-hop з контекстною дедукцією
- **Розуміння зв'язків**: Локальне (в межах батьківського чанку)
- **Контекст**: Повний батьківський контекст

### Приклад
```
Запит: "Як зміна курсу валют впливає на ціни?"

Parent-Child знайде:
- Дочірній чанк: "Курс EUR/USD зріс на 5%"
- Батьківський чанк: Весь розділ про macroeconomic factors ✓
- Контекст включає всю секцію про економічні фактори
- Але все ще може не бачити прямий причинно-наслідковий зв'язок
```

---

## 4. GRAPHRAG: Революційний Підхід

### Механізм Роботи
- **Екстрагує знання** з документів у вигляді **графу**:
  - **Вузли**: Сутності (EUR, USD, ціни, компанії, люди)
  - **Ребра**: Зв'язки (впливає_на, змінюється_через, регулюється)
- Будує **ієрархічні структури** через community detection
- Пошук здійснюється через **граф-обхід** (traversal)
- Здатний до **multi-hop reasoning** (багатошагового міркування)

### Характеристики
- **Структура даних**: Граф знань з ієрархіями
- **Тип пошуку**: Multi-hop (багато стрибків через відносини)
- **Розуміння зв'язків**: Глибоке і багатошарове
- **Контекст**: Повна семантична мережа

### Приклад
```
Запит: "Як зміна курсу валют впливає на ціни?"

GraphRAG знайде:
Вузли:
  - EUR, USD (валюти)
  - Ціни (товарів, послуг)
  - Компанії (експортери, імпортери)
  - ЦБ (центральний банк)

Ребра (зв'язки):
  - EUR →[впливає_на]→ Ціни_експорту
  - USD →[змінюється_через]→ ЦБ_рішення
  - Ціни_експорту →[впливає_на]→ Конкурентоспроможність

Результат: Повна мережа причинно-наслідкових зв'язків ✓
```

## Ключові Відмінності По Аспектам

### 1. Архітектура Даних

**Naive RAG:**
```
Документ → Чанки (512 символів) → Вектори → Векторна БД
```

**Sentence-Window:**
```
Документ → Речення → Вектори + Контекстні вікна → Векторна БД
```

**Parent-Child:**
```
Документ → Батьківські чанки
          ↓
          Дочірні чанки → Вектори (дочірні) → Векторна БД
                          Ієрархія батько-дитина
```

**GraphRAG:**
```
Документ → NER + Relation Extraction → Сутності + Зв'язки
           ↓
Граф знань (вузли, ребра) → Community Detection → Ієрархічний граф
           ↓
Граф + Векторні вкладення → Гібридна схема індексування
```

### 2. Тип Міркування

**Naive & Sentence-Window & Parent-Child:**
- Однокроковий пошук (single-hop)
- "Знайти релевантний документ/речення/чанк"
- Потім LLM міркує на основі цього фрагменту

**GraphRAG:**
- Багатокроковий пошук (multi-hop)
- "Знайти сутність A → слідувати зв'язку → знайти сутність B → слідувати іншому зв'язку → знайти C"
- LLM отримує повну мережу причинно-наслідкових зв'язків

### 3. Якість на Різних Типах Запитів

**Прості запити ("Хто розробив теорію відносності?"):**
- Naive: ✅ 85% точності
- Sentence-Window: ✅ 88%
- Parent-Child: ✅ 90%
- GraphRAG: ✅ 95% (надмірно, але надійно)

**Складні запити ("Як макроекономічні коливання впливають на стратегію розвитку компанії через ланцюг постачання?"):**
- Naive: ❌ 30% точності
- Sentence-Window: ❌ 35%
- Parent-Child: ⚠️ 55%
- GraphRAG: ✅ 87% (справляється чудово)

### 4. Вимоги до Даних

**Naive RAG:**
- Просто: Будь-які текстові документи

**Sentence-Window:**
- Вимагає: Документи зі структурою речень
- Проблеми: З таблицями, кодом, структурованими даними

**Parent-Child:**
- Вимагає: Правильна структура параграфів/розділів
- Проблеми: З неструктурованим текстом

**GraphRAG:**
- Вимагає: **Високоякісна екстракція знань**
  - Вибір правильних сутностей
  - Правильна ідентифікація зв'язків
  - Розуміння контексту для екстракції
- Проблеми: Якість видобутих знань критична

## Практичні Сценарії Вибору

### Сценарій 1: Техпідтримка (FAQ Chatbot)
```
Запити: "Як перезавантажити пристрій?"
Складність: LOW

✅ Рекомендація: Naive RAG або Sentence-Window
❌ GraphRAG: Надмірно складний для цього
```

### Сценарій 2: Корпоративна Вікі (Internal Knowledge Base)
```
Запити: "Які процеси впливають на час доставки?"
Складність: MEDIUM

✅ Рекомендація: Sentence-Window або Parent-Child
⚠️ GraphRAG: Можна, але дорого
```

### Сценарій 3: Фінансовий Аналіз
```
Запити: "Як геополітичні ризики, інфляція та курси валют впливають на бізнес?"
Складність: HIGH (multi-hop reasoning)

✅ Рекомендація: GraphRAG
❌ Інші підходи: Неспроможні дати якісну відповідь
```

### Сценарій 4: Медичне Дослідження
```
Запити: "Які взаємодії ліків найбільш загрозливі для пацієнтів із діабетом 2 типу?"
Складність: HIGH (багатомірні зв'язки)

✅ Рекомендація: GraphRAG
❌ Інші: Можуть пропустити критичні взаємодії
```

### Сценарій 5: Наукові Відкриття
```
Запити: "Які гени впливають на метаболізм через мікробіом кишечника?"
Складність: VERY HIGH (сложные причинно-следственные связи)

✅ Рекомендація: GraphRAG з науковим графом знань
❌ Інші: Недостатньо для першопроходців
```

## Висновки: Яка Система Для Вас?

### Вибирайте **Naive RAG** якщо:
- Use-case має прості, однозначні запити
- Потребуєте максимальної швидкості розгортки
- Обмежені бюджет і ресурси
- Документи без складних взаємозв'язків

### Вибирайте **Sentence-Window** якщо:
- Хочете покращити якість без надмірної складності
- Важливі локальні зв'язки між реченнями
- Потребуєте хорошого балансу простота/якість
- Документи мають природну структуру

### Вибирайте **Parent-Child** якщо:
- Потребуєте вищої якості на складних запитах
- Важлива гнучкість редагування чанків
- Готові інвестувати 5-7 днів на розгортку
- Документи мають ясну ієрархічну структуру

### Вибирайте **GraphRAG** якщо:
- Запити потребують multi-hop reasoning
- Дані виявляють складні, багатомірні зв'язки
- Точність критична (87%+ vs 60-70%)
- Готові інвестувати 2-4 тижні на знання-інженерію
- Enterprise-grade системи (фінанси, медицина, наука)
- Hallucination rate мають бути мінімальними

### Рекомендуємо **Гібридний підхід** якщо:
- Мешта комбінувати Vector + Graph
- Різні типи запитів потребують різних стратегій
- Хочете масштабованість з якістю
