# Порівняння трьох підходів до побудови RAG: Naive, Sentence-Window та Parent-Child Retrieval

## Naive RAG

**Концепція:** Найпростіший та найбільш прямолінійний підхід до RAG. Система розділяє документи на фіксовані за розміром чанки, конвертує їх в векторні вкладення та зберігає у векторній базі даних.

**Механізм роботи:**
- Документи розбиваються на чанки фіксованого розміру (наприклад, 512 символів)
- Кожний чанк конвертується в числовий вектор за допомогою моделі embedding
- Вектори зберігаються у векторній базі даних
- При запиті система знаходить найбільш схожі чанки через косинусну подібність
- Знайдені чанки передаються LLM для генерації відповіді

**Переваги:**
- Простота реалізації (можна налаштувати за кілька днів)
- Відсутність потреби у fine-tuning моделей
- Низькі обчислювальні витрати
- Швидка розгортка в production
- Хорошо працює для простих use-case сценаріїв

**Недоліки:**
- Інформація часто розбивається поперек чанків (наприклад, число розташоване на границі двох чанків)
- Втрата контексту через занадто малі чанки
- Розріджені векторні вкладення при малому розмірі чанків
- Складність пошуку в документах зі складною структурою
- Низька якість на складних питаннях

## Sentence-Window Retrieval

**Концепція:** Розширена техніка, яка балансує точність пошуку з контекстною багатістю. Система обчислює вкладення для окремих речень, але під час вилучення повертає розширене контекстне вікно навколо релевантного речення.

**Механізм роботи:**
- Документи розбиваються на речення як семантичні одиниці
- Вкладення обчислюється для кожного речення окремо
- При збереженні система запам'ятовує розширене вікно: наприклад, одне речення перед та два речення після основного
- При пошуку вкладення запиту порівнюється з векторами окремих речень
- Якщо речення найбільш релевантне, повертається все контекстне вікно (темно-зелене речення + світло-зелені речення з контекстом)

**Переваги:**
- Вищої якості вкладення за рахунок зосередження на окремих реченнях
- Повніший контекст у відповіді за рахунок розширеного вікна
- Вирішення проблеми обраних посилань (pronoun resolution): система розуміє контекст виказувань
- Уникнення фрагментування інформації на границях чанків
- Менше семантичного розрідження ніж при великих чанках

**Недоліки:**
- Вища складність реалізації
- Більше обчислювальних ресурсів для обробки більшої кількості чанків
- Потребує якісної моделі розбиття на речення
- Складність в роботі з нетекстовими структурами (таблиці, списки)
- Потенційно багато надлишкового контексту у вікні

## Parent-Child Retrieval

**Концепція:** Ієрархічний двошаровий підхід, який вирішує дилему між точністю та контекстною повнотою. Система використовує малі чанки (дочірні) для пошуку та великі чанки (батьківські) для контексту.

**Механізм роботи:**
- Документи спочатку розбиваються на великі "батьківські" чанки (параграфи або цілі документи)
- Батьківські чанки далі розбиваються на більш дрібні "дочірні" чанки (одиничні речення або фрази)
- Вкладення обчислюється лише для дочірніх чанків
- При пошуку система знаходить релевантні дочірні чанки
- Повертаються пов'язані батьківські чанки, які містять знайдені дочірні

**Переваги:**
- Оптимальний баланс: точність пошуку + повний контекст
- Максимальна гнучкість: можна редагувати дочірні та батьківські чанки окремо
- Дочірні чанки можуть працювати як спеціалізовані теги для пошуку
- Краще для складноструктурованих документів
- Можливість динамічної регенерації дочірніх чанків при зміні батьківських

**Недоліки:**
- Найвища складність реалізації та налаштування
- Потребує ретельного вибору розміру батьківських/дочірніх чанків
- Більш вибаглива до обслуговування
- Вищі витрати на зберігання та обробку
- Потребує досвіду для оптимального налаштування

## Рекомендації щодо вибору

**Вибирайте Naive RAG якщо:**
- Потребуєте швидкої розгортки прототипу
- Працюєте з простими, однозначними питаннями
- Обмежені обчислювальні ресурси
- Документи мають простої структури
- Бюджет мінімальний

**Вибирайте Sentence-Window якщо:**
- Потребуєте покращення якості без надмірної складності
- Важливі посилання та контекст між реченнями
- Хочете баланс між простотою та якістю
- Документи мають природну структуру на основі речень

**Вибирайте Parent-Child Retrieval якщо:**
- Потребуєте найвищої якості результатів
- Працюєте зі складноструктурованими документами
- Потребуєте гнучкості в редагуванні чанків
- Готові інвестувати в налаштування та обслуговування
- Важливі повні контекстні відповіді
